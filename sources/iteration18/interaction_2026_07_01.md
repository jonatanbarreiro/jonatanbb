## What the code pins down (and why it reshapes the options)

Three facts from the current source decide most of this:

**1. The palette is inverted from the usual "spotlight" case.** Body text is warm near-black serif (`--ink #1c1a16`) on light alabaster paper (`--photo #f5f4f1`). The b's and ring letters glow *gold* because they're a mark floating over the page — but gold-on-light is low-contrast, so for **body** text "lit" has to mean a hot **dark amber / scorched-bronze** that stays legible, brightening toward gold only at the very core of a light where it's transient. This one fact also kills the cheapest trick (see "rejected" below).

**2. The project has already run this experiment, twice, and learned the cost curve.** You have two working precedents that bracket the tradeoff:
- *Ring message* — **real DOM letters recolored per frame** by sampling the light on their arc (`incandescent(level)`), "costs almost nothing." Cheap; but each letter is one uniform color.
- *The b's* — **canvas gradient bands** with true within-element gradients and even direction (the raking), but only affordable because you hand-decomposed the geometry.

And the decisive one: you **deleted the supersampled full-viewport letters *canvas*** because it was "the page's single heaviest per-frame paint." So full-page text rasterization is already empirically off the table. The lesson points one way: **recolor or mask DOM text — never rasterize it.**

**3. The lights are already computed for free.** Every frame `draw()` builds `inn`/`ext` with each point's viewport `px,py`, wake `w`, `gate`, and `glowR ≈ 75·scale` reach. The scattered exterior points are exactly the ones that roam over body text as you scroll. Any text-lighting reuses these — **no new light math** — and slots into the existing discipline: cache page-space rects (like `layoutDots`), no per-frame layout reads, write a style **only when its value changes** (`litCache`/`letterLvl`/`letterTf`).

## The constraint that makes or breaks all of it: culling

Performance here is **not** about the per-element technique — it's about *how many elements are touched per frame*. At any instant only a handful of text elements sit within `glowR` of a light. So: cache each candidate element's page-rect, each frame test distance to the few nearby lights, and only the handful in reach do any work; everyone else is reset to plain ink **once** and then skipped. Restrict the candidate set to the headline-ish text (hero name, lede, kickers, entry titles) — small glyph counts, tasteful, bounded. Done this way the marginal cost is a few elements, never the page. Every approach below lives or dies on this, and it's the same discipline already in the file.

## The approaches, ranked

**① Text as a window onto a moving light field — `background-clip:text` + radial gradients (recommended core).** Each lit element's *fill* becomes `radial-gradient(circle glowR at Lx Ly, amber, transparent)` (one stop-stack per nearby light) over an ink base; `color: transparent; background-clip: text`. This is your idea **(a)** made *continuous* (a real gradient, not 5 discrete annuli) and your idea **(c)** made *robust* (one element, no font-weight assumptions) at the same time. Lights only the glyphs, never the paper; no per-glyph geometry; GPU-composited; reuses the `incandescent` ramp.
